# Governance

This document describes how CNMSQL is run: who makes decisions, how those decisions get made, and how people take on more responsibility over time.

## Roles

### Contributors

Anyone who files an issue, reviews a pull request, writes documentation, or sends a patch is a contributor. There is no formal step to join. Read the [contributing guide](CONTRIBUTING.md) and get started.

### Maintainers

Maintainers carry write access and the responsibility that comes with it. They review and merge changes, triage issues, cut releases, and set technical direction. The current list lives in [MAINTAINERS.md](MAINTAINERS.md).

A maintainer is expected to:

- Review pull requests in their area and help contributors land changes.
- Keep the project healthy: triage issues, watch CI, and respond to security reports.
- Uphold the [Code of Conduct](CODE_OF_CONDUCT.md).

## Becoming a maintainer

Maintainers are drawn from contributors who have shown sustained, high-quality involvement: solid patches, careful reviews, and good judgment about the project's direction. There is no fixed quota of commits. When an existing maintainer believes a contributor is ready, they propose it to the others, and the change is made by consensus.

A maintainer who steps back from active work may move to emeritus status. This is a normal and respected outcome, not a demotion.

## Making decisions

Most decisions happen in the open on issues and pull requests and need no ceremony. When a question needs a deliberate call, maintainers decide by lazy consensus: a proposal is made, and if no maintainer objects within a reasonable window, it carries. If there is disagreement that discussion can't resolve, a simple majority of maintainers settles it.

Changes to this governance document, to the list of maintainers, or to the project license follow the same consensus process among maintainers.

## Code of Conduct

Everyone in the project's spaces is bound by the [Code of Conduct](CODE_OF_CONDUCT.md). Maintainers are responsible for enforcing it.

## Changing this document

This document evolves with the project. Propose changes through a pull request, and the maintainers decide by the process above.
